\#\# 1\. Executive Summary & Project Objectives  
\[cite*\_start\]**\*\*LibreLog (Rumble)\*\*** is an open-source, web-based business, traffic, and programming scheduling system designed for community and independent radio stations.* 

*\[cite\_*start\]The primary goal is to democratize the digital airwaves by providing a modern, accessible tool to manage client contracts, underwriting, 24-hour music clock rotations, and browser-based voice tracking without requiring expensive proprietary Windows software\[cite: 13, 15, 18, 47\].

\#\#\# System Architecture Topology  
\[cite\_start\]The project operates on a decoupled **\*\*Scheduler $\\rightarrow$ Studio $\\rightarrow$ Transmitter\*\*** operational design\[cite: 105, 108\]:  
\* \[cite*\_start\]**\*\*Rumble (LibreLog):\*\*** The business and scheduling controller (Java Spring Boot \+ React)\[cite: 49, 92\].*  
*\* \[cite\_*start\]**\*\*Jazz (LibreTime):\*\*** The studio automation and physical playout engine\[cite: 104, 110\].  
\* \[cite*\_start\]**\*\*Banshee (AzuraCast):\*\*** The transmitter, handling Icecast mounts, streaming endpoints, and listener analytics\[cite: 92, 104\].*

\#\# 2\. Infrastructure & Integration Bridge (DevOps / Systems Spec)

Rumble operates on a separate server from Jazz to ensure developmental isolation and uptime protection\[cite: 134, 158, 160\].

\#\#\# DB-01: Jazz Database Interconnect  
Rumble queries Jazz’s underlying PostgreSQL database (\`libretime\` / \`airtime\`) directly via read-only channels\[cite: 38, 51, 140, 162\].  
\* \*\*Jazz Configuration (\`postgresql.conf\`):\*\* Set \`listen\_addresses \= 'localhost,\[Jazz\_IP\]'\`\[cite: 161\].  
\* \*\*Jazz Authentication (\`pg\_hba.conf\`):\*\* Append the following line to allow authentication from Rumble\[cite: 162\]:  
  \`\`\`text  
  host    libretime    libretime\_user    \[Rumble\_IP\]/32    md5

### **FS-01: SFTP File Transport**

* Audio files (voice tracks, commercials, promos) generated by Rumble are transferred to Jazz using an SSH Keypair via SFTP.  
* Target landing folder on Jazz: /srv/libretime/rumble\_import/.

### **API-01: Automated Library Ingestion**

* After an SFTP upload completes, Rumble executes an HTTP API request to Jazz's internal API (using the key configured in /etc/libretime/config.yml) or executes a CLI import command to force immediate library indexing.

## **4\. Scheduling Engine & Burnout Logic Specification**

Rumble generates a complete 24-hour daily log based on hourly Format Clocks and rotational rules:

### **Rotational Burnout Rules**

* **Artist Separation (SCH-03):** The scheduling algorithm must verify that the same artist is not scheduled within **90 minutes** of a prior play.  
* **Song Separation (SCH-03):** The scheduling algorithm must verify that the exact same song is not repeated within **4 hours (240 minutes)**.  
* **Clutter Control:** Underwriting spots from competing clients (or the same client) must not be scheduled back-to-back within the same commercial break.

## **5\. Broadcast Audio Processing Pipeline (Java / FFmpeg)**

To prevent dynamic volume jumps and glitches, all user-submitted audio and browser voice tracks must be standardized by Rumble before being transferred to Jazz.

### **Processing Rules:**

1. **Encoding Standard (AUD-02):** **MP3, 44.1 kHz, 192kbps Constant Bitrate (CBR), Stereo**.  
2. **Headroom Ceiling (AUD-03):** Peak normalization pass capped at **\-2.0 dBFS**. This leaves dynamic headroom for Jazz's built-in ReplayGain process to normalize target loudness to **\-14 dBFS**.  
3. **Metadata Tags (AUD-04):** Programmatically write clean UTF-8 ID3 tags prior to upload (e.g., Artist: LibreLog VT, Title: VT-\[Host\]-\[Date\]-\[Hour\]).

## **6\. Voice Tracking Suite (React Frontend & API)**

Allows hosts and DJs to record linking voice segments remotely via the web browser:

1. **Timeline Grid UI:** Displays a 24-hour visual grid with explicit placeholder indicators ("Empty VT Slots") between tracks.  
2. **Browser Recording (HTML5 Web Audio API):** Captures raw audio via the browser microphone.  
3. **Overlap Markers:** Enables DJs to set crossfade/ducking points (outro fade of Track A vs. intro start of Track B).  
4. **Backend Ingestion:** React uploads raw WebM/WAV to Java $\\rightarrow$ Java transcodes to MP3 $\\rightarrow$ Transfers to Jazz via SFTP $\\rightarrow$ Triggers import API $\\rightarrow$ Updates cc\_schedule playlist sequence.

## **7\. System Orchestration & Safety Rules**

1. **Lock Window Rule:** Rumble's scheduling engine is strictly prohibited from modifying the active playing hour or $T+1$ hour in Jazz's database.  
2. **Injection Window:** Schedule changes or logs must be written **24 hours in advance** (or at minimum $T+2$ hours out) to ensure playout buffers are stable.  
3. **As-Run Reconciliation Loop:** A nightly cron job queries Jazz’s cc\_playlist\_anal execution history table to cross-verify played assets against Rumble's campaign requirements, logging unfulfilled spots for make-up passes.

## Format Clocks v2 — Hot-Clock Timing Engine (implemented v2.7.0)

Clocks now support professional hot-clock semantics on the live v2 model (the
`rumble_format_clock*` scaffolding tables remain unused):

* **Anchors** — any slot may be pinned to `@mm:ss` from the show-instance start
  (legal ID at 0:00, avail A at 18:00). `SOFT` anchors start late and flag;
  `HARD` anchors trim the preceding music/pad via LibreTime `cue_out` (floor 20s,
  smooth fade) so the anchor lands exactly.
* **Back-timing** — every hour is fitted plan-then-write by the segment planner
  (`ClockSegmentPlanner`): underruns are padded from the station's configured
  **pad/sweeper cart** (fallback: first IMAGING cart), the final music/pad unit is
  cue-out-trimmed so the instance ends exactly at the top of the hour; non-music
  content (spots, interviews) is never cut — overruns are flagged instead.
* **Commercial avails** — a fill block may carry both a unit count and a
  total-seconds cap ("max 3 spots / 120s"); expanded units share a `fill_group`
  and the cap is enforced across the group at push (15s grace).
* **Preview = plan** — Preview push runs the same planner in dry-run (no writes,
  no rotation-pointer persistence) and shows planned air times, trims, pads and
  anchor misses.
